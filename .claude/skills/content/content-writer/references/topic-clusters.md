# Topic Clusters & Topical Authority Strategy

This is the reference for planning content that builds search authority through interconnected topic clusters, not isolated pieces. Read this when the user wants to plan a content strategy, build topical authority, or move beyond one-off blog posts.

**The principle:** Search engines reward sites that demonstrate comprehensive coverage of a topic. A single page rarely ranks alone. A cluster of interconnected pages, anchored by a strong pillar, signals authority better than any single post.

---

## 1. What Is a Topic Cluster?

A topic cluster is a content architecture with three layers:

```
                    ┌──────────────────┐
                    │   PILLAR PAGE     │ (broad, comprehensive)
                    │  Covers topic X   │
                    └────────┬─────────┘
                    ┌────────┴──────────┐
                    │  CLUSTER CONTENT   │
                    │   (subtopic deep   │
                    │    dives, each     │
                    │    linking to      │
                    │    the pillar)     │
                    └────────┬──────────┘
                    ┌────────┴──────────┐
                    │  SUPPORTING       │
                    │  CONTENT          │
                    │  (social, email,  │
                    │   video, etc.)    │
                    └───────────────────┘
```

| Layer | What it is | Job | Example |
|---|---|---|---|
| **Pillar Page** | Long-form, broad coverage of a core topic | Own the topic in search; the hub | "The Complete Guide to SaaS Metrics" |
| **Cluster Content** | Deep dives on specific subtopics | Capture long-tail queries; build specific authority | "How to Calculate MRR Churn," "What Is LTV:CAC Ratio?," "Net Revenue Retention Explained" |
| **Supporting Content** | Other formats that promote/distribute | Drive traffic, earn links, build awareness | Social posts, email sequences, guest posts, videos |

Each cluster content piece links **to** the pillar page, and the pillar page links **to** each cluster piece (or relevant sections of cluster pieces).

---

## 2. Cluster Structure Decision

### Spoke (hub-and-spoke) model
One pillar + many cluster pages. Best when the topic is broad enough to support 10+ subtopics.

```
              ┌─── subtopic A
              │
    Pillar ───┼─── subtopic B
              │
              ├─── subtopic C
              └─── subtopic D
```

### Pyramid model
One pillar, multiple sub-pillars, each with its own cluster pages. Best for very broad topics with sub-categories.

```
              ┌─── Pillar
              │
        ┌─────┼─────┐
        │     │     │
    Sub-    Sub-   Sub-
    pillar  pillar pillar
    │││     │││    │││
    cluster pages per sub-pillar
```

### Linear (content hub) model
No single pillar: a hub page indexes multiple related pieces. Best when the content doesn't center on a single comprehensive topic.

```
    ┌────────────────────┐
    │   Content Hub      │ (index/navigation page)
    ├────────────────────┤
    │ Topic 1 → article  │
    │ Topic 2 → article  │
    │ Topic 3 → article  │
    └────────────────────┘
```

## 3. How to Build a Topic Cluster

### Step 1: Choose the core topic
- Pick a topic that is broad enough to support 8-15 cluster pieces.
- Must be relevant to your business and have search demand.
- Must be something you can cover with genuine depth and authority.
- Test: If you can't write 10 subtopic questions, the topic is too narrow.

### Step 2: Research cluster subtopics
Sources for subtopic ideas:
- "People Also Ask" boxes for your core topic.
- Keyword research tools: filter for long-tail keywords under the core topic.
- Competitor site structure: what sub-pages do they have under this topic?
- Customer questions from support tickets, sales calls, onboarding.
- Wikipedia table of contents for the topic.
- Existing content you can expand or reorganize.

### Step 3: Write the pillar page
- 2,500-5,000+ words covering the topic comprehensively.
- Use H2s for major sections that map to cluster topics.
- Include a table of contents linking to each section.
- Provide the "big picture" overview with enough depth to stand alone.
- Internal link to existing cluster pieces where they add depth.
- Leave room for new cluster pieces: sections that say "covered in depth elsewhere."

### Step 4: Write cluster pieces
- 1,000-2,500 words each, depending on the subtopic.
- Each answers a specific question or covers a specific angle.
- Link naturally to the pillar page from the body content.
- Use descriptive anchor text: not "click here," but "our full guide to SaaS metrics covers this in more depth."

### Step 5: Interlink strategically
- Each cluster page → links to the pillar page (1-2 natural links).
- Pillar page → links to each cluster page (usually from the relevant section).
- Cluster pages → link to each other when genuinely relevant.
- Don't force links. If it doesn't help the reader, don't link.

### Step 6: Promote and iterate
- Build links to the pillar page to boost cluster authority.
- Update cluster pieces periodically to maintain freshness signals.
- Monitor rankings for the core topic and expand the cluster where gaps remain.
- Add new cluster pieces as new subtopics emerge.

## 4. Pillar Page Structure

A strong pillar page typically follows this structure:

```
TITLE: "The Complete Guide to [Topic] (Year)"

[Brief intro: what this guide covers and who it's for]

## Table of Contents
[Links to all major sections]

## [Section 1: The fundamentals / what is X]
[2-3 paragraphs. Define the topic. Set the context.
 Link to relevant cluster content for deeper exploration.]

## [Section 2: Why it matters]
[Stakes and importance. For the skimmer who needs to know
 "why should I care?"]

## [Section 3: Key components / types / approaches]
[The taxonomy of the topic. List the major sub-elements.
 Each one is a potential cluster piece.]

## [Section 4 through N: Deep sections]
[3-6 major sections covering the breadth of the topic.
 Each section should:
 - Be self-contained (the reader can read this section alone)
 - Link to the relevant cluster piece for depth
 - Include at least one concrete example]

## [Implementation guide / how to apply this]
[Practical steps, frameworks, or frameworks. Ground the theory.]

## [Common mistakes / pitfalls]
[Acknowledges what the other guides leave out. Adds credibility.]

## [FAQs]
[Schema-marked-up FAQ section. Capture voice search traffic.]

## [Glossary / key terms]
[Optional: useful for technical topics.]

## [Resources / next steps]
[CTA, related guides, further reading.]
```

## 5. Cluster Content Ideas by SubTopic Depth

| Subtopic type | Angle | Word count |
|---|---|---|
| **Definition explainer** | "What is [subtopic]?" | 800-1,200 |
| **How-to** | "How to [do subtopic] | 1,200-2,000 |
| **Comparison** | "[X] vs [Y] for [subtopic]" | 1,000-1,500 |
| **Myth-buster** | "[Number] myths about [subtopic]" | 800-1,200 |
| **Data-driven** | "[Number] [subtopic] statistics" | 1,000-1,500 |
| **Case study** | "How [company] did [subtopic]" | 800-1,500 |
| **Beginner guide** | "[Subtopic] for beginners" | 1,200-2,000 |
| **Advanced guide** | "Advanced [subtopic]: [specific technique]" | 1,500-2,500 |
| **Pitfalls** | "[Number] [subtopic] mistakes to avoid" | 800-1,200 |
| **Checklist/template** | "[Subtopic] checklist / template" | 600-1,000 |

## 6. Interlinking Best Practices

### Pillar → Cluster
- Link from the section where the cluster topic is relevant.
- Use H2/H3 context to make the link feel natural.
- Anchor text: descriptive phrase, not the full title.

### Cluster → Pillar
- Link contextually within the body, not just in a "related posts" widget.
- The anchor should reference the broader context: "for a full overview, see our complete guide."
- Avoid forcing the link: if it doesn't fit naturally, put it in the "related resources" section.

### Cluster ↔ Cluster
- Only link cluster-to-cluster when the connection is genuinely useful.
- A cluster about "MRR Churn" can naturally link to a cluster about "Net Revenue Retention."
- Don't link cluster pieces just for the sake of internal linking.

### Linking density
| Page type | Suggested internal links |
|---|---|
| Pillar page | 5-15 links (to cluster pieces and other relevant content) |
| Cluster page | 2-5 links (to pillar, 1-2 related clusters, 1-2 other relevant posts) |
| Supporting content | 1-3 links (to pillar or relevant cluster) |

## 7. Measuring Cluster Performance

| Metric | What it tells you |
|---|---|
| **Pillar page ranking** for core topic | Is the cluster working as a whole? |
| **Cluster page rankings** for subtopics | Are individual pieces doing their job? |
| **Pillar page traffic** over time | Is topical authority growing? |
| **Internal link click rate** | Are readers navigating the cluster? |
| **Time on page** for pillar | Is the content engaging at the top level? |
| **Bounce rate** from cluster pages | Are subtopics matching search intent? |
| **Featured snippet wins** | Has your cluster earned position zero? |
| **Backlinks to pillar + cluster** | Is the cluster earning external authority? |
| **Conversions from cluster** | Is the cluster generating business results? |

---

## 8. When NOT to Use Topic Clusters

Topic clusters aren't always the right approach. Skip them when:
- **You have <10 pieces of content**: build individual pieces first, then cluster later.
- **The topic is too narrow**: one strong piece may be enough.
- **The goal is time-sensitive**: clusters take months to build authority. A single strong piece may be faster.
- **The audience doesn't search**: if your audience comes from referrals, social, or direct, invest elsewhere.
- **You can't commit to quality across all pieces**: one weak cluster piece can drag down the perception of the whole cluster.

## 9. Example: SaaS Metrics Topic Cluster

### Pillar
"The Complete Guide to SaaS Metrics (2026)" - 4,000 words covering all key metrics.

### Cluster pieces
1. "What Is MRR? How to Calculate Monthly Recurring Revenue" (1,500 words)
2. "Net Revenue Retention vs Gross Revenue Retention: What's the Difference?" (1,200 words)
3. "How to Calculate Churn Rate (And What's a Good Churn Rate by Industry)" (1,800 words)
4. "LTV:CAC Ratio: What it's and Why It Matters More Than Revenue" (1,400 words)
5. "SaaS Payback Period: How to Calculate and Improve It" (1,200 words)
6. "What Is Burn Multiple? The Startup Efficiency Metric Explained" (1,000 words)
7. "ARR vs MRR: When to Use Each Revenue Metric" (1,000 words)
8. "CAC Payback Period Benchmarks by SaaS Segment" (1,300 words)
9. "12 SaaS Metrics Every Board Report Should Include" (1,500 words)
10. "The Difference Between Real-Time and Historical SaaS Analytics" (1,100 words)

### Interlink pattern
- Pillar has a table of contents. Each section link goes to the cluster piece.
- Each cluster piece links back to the pillar from the first mention of "SaaS metrics."
- Cluster pieces about related metrics (NRR vs GRR) link to each other.
- "ARR vs MRR" and "What Is MRR" link to each other as related reads.

### Supporting content
- Email series: "5 Days of SaaS Metrics" (5 emails, each spotlighting one cluster piece)
- LinkedIn posts: one metric explained per post, linking to the pillar
- YouTube shorts: "Explain [Metric] in 60 Seconds" (10 videos)
- Guest post on a SaaS publication: "The 3 SaaS Metrics Most Founders Get Wrong" (links to pillar)
